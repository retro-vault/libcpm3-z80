libcpm3-z80

CP/M 3 Standard C Library for Z80
Maintained by retro-vault

Abstract

libcpm3-z80 is a portable, readable, and minimal standard C library for CP/M 3 on Z80. It targets SDCC, keeps platform-dependent code isolated, and exposes a small but test-backed subset of ISO C together with a CP/M-oriented Unix-like I/O layer.

Introduction

libcpm3-z80 is an attempt to provide a library written in straightforward C, with a clear separation between platform-independent code and the small amount of platform-specific support that CP/M 3 cannot provide through BDOS alone.

The implementation is now organized directly by public subsystem under src/: stdlib/, stdio/, string/, sys/, platform/, setjmp/, and related folders. The older src/_impl/ layout is gone.

Building the Library

Clone the repository first: 

git clone https://github.com/tstih/libcpm3-z80.git
cd libcpm3-z80

The supported build commands are:

Command Description
make Build the library.
make test Build the library and tests, then run them in the CP/M emulator.
make clean Remove build/ and bin/.

Build parameters:

Parameter Values Default Description
DOCKER on, off on on builds inside the wischner/sdcc-z80 image; off builds natively and requires SDCC on PATH.
PLATFORM name none Defines PLATFORM_<NAME> for conditional platform code. The default platform provides msleep() and _libinit().
BUILD_DIR path build/ Intermediate build products.
BIN_DIR path bin/ Final outputs: library, CRT0, headers. 
make
make DOCKER=off
make PLATFORM=myboard
make DOCKER=off BUILD_DIR=out/build BIN_DIR=out/bin
make test requires Docker. With DOCKER=off the test binaries can be built, but they cannot be run through the emulator path.

Build outputs are placed in bin/ unless BIN_DIR is overridden:

File Description
crt0cpm3-z80.rel C runtime start-up object. Link this first.
libcpm3-z80.lib CP/M 3 standard C library archive.
include/ Public header files copied into the output directory.

libsdcc-z80.lib is downloaded to lib/ on the first make test and must also be linked into programs that use this library.

The library targets SDCC Z80 __sdcccall(1). Use SDCC version 4.2.0 or newer.

Compiling Your CP/M Program

You can use a local SDCC installation directly, or adapt the Docker invocation from test/Makefile. The link order is fixed: CRT0 first, your object files next, then libcpm3-z80.lib, and finally the SDCC runtime stubs. 

sdcc -mz80 --std-c11 --no-std-crt0 --nostdinc --nostdlib \
     -I bin/include -c -o myprog.rel myprog.c

sdcc -mz80 --std-c11 --no-std-crt0 --nostdinc --nostdlib \
     --code-loc 0x100 --data-loc 0 \
     -o myprog.ihx \
     bin/crt0cpm3-z80.rel myprog.rel \
     bin/libcpm3-z80.lib lib/libsdcc-z80.lib

sdobjcopy -I ihex -O binary myprog.ihx myprog.com
Practical rule: CRT0 must be first on the link line, and plain upstream SDCC must be told explicitly to use -mz80, --code-loc 0x100, and --data-loc 0 for CP/M .COM programs.

The project's Docker toolchain already defaults to Z80, but the memory layout requirements still matter if you invoke upstream sdcc yourself.

Running the Tests

Run the test suite with: 

make test

This builds the library and all test .com programs inside Docker, then runs them automatically in a RunCPM CP/M 3 emulator container. Results are written to bin/<name>.txt.

Binary Coverage
tctype.comctype.h character classification
tstring.comstring.h string functions
tstdlib.comstdlib.h general utilities
tstdio.comstdio.h file I/O
ttime.comtime.h time functions
tmath.commath.h floating-point math
tmem.comallocator internals and utility lists
tfile.comlow-level file API
tsdcc.comSDCC runtime integration
tsetjmp.comsetjmp/longjmp

Each program prints PASS or FAIL per test case and ends with a summary line. The produced .com files can also be copied to a real CP/M disk or run under another emulator.

Source Tree Layout

Implementation files are grouped by the public library area they support:

Folder Contents
src/sys/CP/M system bindings such as bdos.s and crt0cpm3-z80.s
src/stdlib/stdlib implementation plus allocator, list, and startup internals
src/stdio/stdio implementation plus internal formatting helpers
src/platform/platform-specific hooks and timing helpers
src/setjmp/setjmp and longjmp
src/string/, src/time/, src/math/, src/file/, ...subsystem implementations

Platform-dependent Functions

Hardware-specific behavior is isolated behind the PLATFORM build parameter. Typical values are none for the built-in default platform and project-specific names such as partner or myboard for custom ports. 

make PLATFORM=none
make PLATFORM=partner
make PLATFORM=myboard

At build time this defines PLATFORM_<NAME> and also exposes the bare token PLATFORM_NAME together with the string form PLATFORM_NAME_STR.

All platform-specific symbols are declared in platform.h. In particular:

Symbol Role
libplatformPlatform name string, usually initialized from PLATFORM_NAME_STR
nltypeNewline translation mode for all console output
prognameOptional program name if your startup code provides one
msleep()Busy-wait delay in milliseconds
_libinit()Platform initializer called once at startup
  • NL_LF: emit \n only.
  • NL_CRLF: emit \r\n, the built-in default for CP/M-style console output.
  • NL_LFCR: emit \n\r.

If you choose a PLATFORM other than none, the library does not provide msleep() or _libinit(); you must supply them yourself and link your object before the library. 

#include <platform.h>

void _libinit(void) {
    libplatform = PLATFORM_NAME_STR;
    nltype      = NL_CRLF;
}

What Is Implemented?

The project currently provides a small, test-backed subset of the standard C library, plus a CP/M-oriented POSIX-like layer and a few project-specific extensions.
